# 数据增量更新功能规范文档

## 1. 概述

### 1.1 目标
优化现有的数据抓取脚本，实现增量更新机制，避免每次全量获取重复数据，减少豆瓣 API 请求次数，提高更新效率。

### 1.2 背景
当前问题：
- 每次运行脚本都要获取所有数据的详细信息（约100条综艺/多条电影热剧）
- `fetchDetailByTitle` 是最耗时的操作，每条数据都要调用
- 即使数据没变化，也会重新获取所有详情
- 大量重复请求浪费时间和资源

### 1.3 预期效果
- 新数据获取详情，已存在数据跳过详情获取
- 请求次数减少 70% 以上
- 更新时间大幅缩短
- 保持数据完整性和热力值准确性

## 2. 技术方案

### 2.1 数据结构设计

#### 2.1.1 索引结构
在 `data.json` 中添加索引字段，用于快速比对：

```json
{
  "variety": [...],
  "varietyIndex": {
    "douban_12345": {
      "title": "向往的生活",
      "rate": "7.6",
      "year": "2017",
      "lastUpdate": "2026-04-12"
    }
  },
  "movieIndex": {...},
  "dramaIndex": {...}
}
```

#### 2.1.2 数据项结构
保持现有结构，添加豆瓣原始ID：

```json
{
  "id": "show_001",
  "doubanId": "26991234",
  "title": "向往的生活",
  "rate": "7.6",
  "cover": "...",
  "year": "2017",
  "hotScore": 165,
  "subCategory": "真人秀",
  ...
}
```

### 2.2 更新流程

```
┌─────────────────────────────────────────────────────────────┐
│                      增量更新流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 加载现有数据和索引                                       │
│           ↓                                                 │
│  2. 获取豆瓣列表数据（只有基本信息，快速）                    │
│           ↓                                                 │
│  3. 比对索引：                                               │
│     - 新增ID → 加入待获取详情队列                            │
│     - 已存在ID → 跳过详情获取，加入更新队列                   │
│           ↓                                                 │
│  4. 批量获取新数据详情                                       │
│           ↓                                                 │
│  5. 合并数据：                                               │
│     - 新数据：完整信息                                       │
│     - 已存在数据：更新热力值、评分                           │
│           ↓                                                 │
│  6. 重新排序、生成索引、保存                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 热力值更新策略

热力值计算公式：`baseScore(100) + timeBonus + rateBonus`

| 场景 | 处理方式 |
|------|----------|
| 新数据 | 完整计算热力值 |
| 已存在数据 | 重新计算热力值（评分可能变化） |
| 定期全量更新 | 每周一次，重新获取所有详情 |

### 2.4 更新模式

| 模式 | 触发条件 | 行为 |
|------|----------|------|
| 增量更新 | 日常运行 | 只获取新增数据详情 |
| 全量更新 | 每周/手动 | 获取所有数据详情 |
| 强制更新 | 参数指定 | 忽略索引，全量获取 |

## 3. 文件修改清单

### 3.1 需要修改的文件

| 文件 | 修改内容 |
|------|----------|
| `scripts/fetch_variety.js` | 添加增量更新逻辑 |
| `scripts/fetch_movie.js` | 添加增量更新逻辑 |
| `scripts/fetch_drama.js` | 添加增量更新逻辑 |

### 3.2 公共模块

创建 `scripts/lib/incremental.js`，封装公共逻辑：
- `loadExistingData()` - 加载现有数据和索引
- `buildIndex(items)` - 构建索引
- `compareWithExisting(listData, existingIndex)` - 比对新旧数据
- `mergeData(newItems, existingItems)` - 合并数据
- `saveData(data, indexPath)` - 保存数据和索引

## 4. API 设计

### 4.1 命令行参数

```bash
# 增量更新（默认）
node scripts/fetch_variety.js

# 强制全量更新
node scripts/fetch_variety.js --full

# 只更新热力值（不获取详情）
node scripts/fetch_variety.js --hot-only
```

## 5. 兼容性

### 5.1 向后兼容
- 首次运行时，没有索引文件，自动执行全量更新
- 索引字段可选，不影响现有数据读取

### 5.2 数据迁移
- 现有 `data.json` 无需迁移
- 首次运行新脚本时自动生成索引

---

## 6. 风险与对策

| 风险 | 对策 |
|------|------|
| 评分变化未及时更新 | 定期全量更新 + 热力值每次重算 |
| 索引损坏 | 索引可重建，不影响主数据 |
| 豆瓣API变化 | 保持原有API调用逻辑不变 |
